02.2.3

Transferencias

Proporciona disponibilidad presupuestal mediante la reasignación de montos entre distintas claves (Suma Cero) sin alterar el techo presupuestal global.

Tipo: Proceso Financiero y Transaccional
1 Descripción General

Módulo transaccional que permite proporcionar o reasignar disponibilidad presupuestal entre distintas claves presupuestales durante el ejercicio activo. Al ser de tipo "compensadas", operan matemáticamente bajo la regla de Suma Cero, donde el monto total de las reducciones (origen) debe ser exactamente igual al monto total de las ampliaciones (destino).

CREATE OR REPLACE FUNCTION presupuesto.fn_abc_transferencias_ppto(
    p_id_transferencia integer DEFAULT NULL,
    p_fecha date DEFAULT NULL,
    p_folio character varying DEFAULT NULL,
    p_justificacion text DEFAULT NULL,
    p_monto_total numeric DEFAULT 0,
    p_pe_ejercicio_trabajo integer DEFAULT NULL,
    p_mes_trabajo smallint DEFAULT NULL,
    p_usuario character varying DEFAULT NULL,
    p_accion integer DEFAULT NULL,
    p_tr_tipo smallint DEFAULT NULL,
    p_pe_origen bigint DEFAULT NULL,
    p_pe_destino bigint DEFAULT NULL,
    p_documento_soporte character varying DEFAULT NULL
)
 RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER AS $function$
 DECLARE 
    v_id integer := p_id_transferencia;
    v_folio character varying := p_folio;
    v_consecutivo integer;
    v_periodo_abierto boolean;
 BEGIN
    -- 1. Validacion de Periodo de Escritura
    IF p_pe_ejercicio_trabajo IS NOT NULL AND p_mes_trabajo IS NOT NULL THEN
        SELECT config.fn_periodo_permite_escritura(p_pe_ejercicio_trabajo, p_mes_trabajo) INTO v_periodo_abierto;
        IF NOT COALESCE(v_periodo_abierto, true) THEN
            RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'El periodo correspondiente al ejercicio/mes no permite escritura.');
        END IF;
    END IF;

    CASE p_accion
        WHEN 2 THEN -- ALTA
            IF v_folio IS NULL OR v_folio = '' OR v_folio = 'S/F' OR v_folio = 'S/O' THEN
                SELECT COALESCE(MAX(SUBSTRING("TR_FOLIO" FROM 6)::integer), 0) + 1
                INTO v_consecutivo
                FROM presupuesto.transferencias_ppto
                WHERE "TR_EJERCICIO" = p_pe_ejercicio_trabajo AND "TR_FOLIO" LIKE 'OAJ/T%';
                v_folio := 'OAJ/T' || LPAD(v_consecutivo::text, 3, '0');
            END IF;

            INSERT INTO presupuesto.transferencias_ppto (
                "TR_EJERCICIO", "TR_FOLIO", "TR_TIPO", "TR_FECHA",
                "TR_PE_ORIGEN", "TR_PE_DESTINO", "TR_IMPORTE", "TR_CONCEPTO",
                "TR_DOCUMENTO_SOPORTE", "TR_ESTATUS", "USUARIO_C", "FECHA_C"
            ) VALUES (
                p_pe_ejercicio_trabajo, v_folio, COALESCE(p_tr_tipo, 1), COALESCE(p_fecha, now()::date),
                p_pe_origen::integer, p_pe_destino::integer, p_monto_total,
                COALESCE(p_justificacion, 'S/O'), COALESCE(p_documento_soporte, 'S/O'),
                1, COALESCE(p_usuario, 'sistema'), now()
            ) RETURNING "TR_NO" INTO v_id;

        WHEN 3 THEN -- CAMBIO
            UPDATE presupuesto.transferencias_ppto 
            SET "TR_IMPORTE" = p_monto_total, "TR_CONCEPTO" = COALESCE(p_justificacion, "TR_CONCEPTO"),
                "TR_PE_ORIGEN" = p_pe_origen::integer, "TR_PE_DESTINO" = p_pe_destino::integer,
                "TR_DOCUMENTO_SOPORTE" = COALESCE(p_documento_soporte, "TR_DOCUMENTO_SOPORTE"),
                "TR_FECHA" = COALESCE(p_fecha, "TR_FECHA"), "TR_TIPO" = COALESCE(p_tr_tipo, "TR_TIPO"),
                "USUARIO_M" = p_usuario, "FECHA_M" = now()
            WHERE "TR_NO" = v_id AND "TR_ESTATUS" = 1;
            IF NOT FOUND THEN RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Registro no encontrado o no modificable.'); END IF;

        WHEN 4 THEN -- BAJA LÓGICA
            UPDATE presupuesto.transferencias_ppto SET "TR_ESTATUS" = 4, "USUARIO_M" = p_usuario, "FECHA_M" = now() WHERE "TR_NO" = v_id;
            IF NOT FOUND THEN RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Registro no encontrado.'); END IF;
    END CASE;

    RETURN jsonb_build_object('cod_resultado', 0, 'msg_resultado', 'OK', 'id_transferencia', v_id, 'TR_FOLIO', v_folio);
 EXCEPTION WHEN OTHERS THEN 
    RETURN jsonb_build_object('cod_resultado', -2, 'msg_resultado', 'Error: ' || SQLERRM);
 END;$function$

CREATE OR REPLACE FUNCTION presupuesto.fn_leer_transferencias_ppto(
    p_id bigint DEFAULT NULL,
    p_folio character varying DEFAULT NULL,
    p_tipo smallint DEFAULT NULL,
    p_pe_ejercicio integer DEFAULT NULL,
    p_estatus smallint DEFAULT NULL,
    p_pe_origen bigint DEFAULT NULL,
    p_pe_destino bigint DEFAULT NULL,
    p_fecha_desde date DEFAULT NULL,
    p_fecha_hasta date DEFAULT NULL,
    p_page integer DEFAULT 1,
    p_limit integer DEFAULT 50,
    p_pe_ejercicio_trabajo integer DEFAULT NULL,
    p_mes_trabajo smallint DEFAULT NULL,
    p_usuario character varying DEFAULT NULL,
    p_accion integer DEFAULT 1
) RETURNS jsonb LANGUAGE plpgsql STABLE SECURITY DEFINER AS $function$
DECLARE 
    v_result JSONB; 
    v_offset INTEGER; 
    v_PE_EJERCICIO INTEGER := COALESCE(p_PE_EJERCICIO_trabajo, config."devuelve_ejercicio"());
BEGIN
    IF p_accion <> 1 THEN RETURN jsonb_build_object('error', 1, 'mensaje', 'Acción no soportada.', 'data', '[]'::jsonb); END IF;
    
    IF p_id < 0 THEN
        RETURN jsonb_build_object('error', 0, 'mensaje', 'OK', 'PE_EJERCICIO', v_PE_EJERCICIO, 'rowcount', 1, 'data', jsonb_build_array(jsonb_build_object('tr_no', NULL, 'tr_folio', NULL, 'tr_PE_EJERCICIO', v_PE_EJERCICIO, 'tr_estatus', 1)));
    END IF;
    
    v_offset := (GREATEST(p_page, 1) - 1) * GREATEST(p_limit, 1);
    
    SELECT jsonb_agg(t) INTO v_result FROM (
        SELECT tr."TR_NO" as tr_no, tr."TR_FOLIO" as tr_folio, tr."TR_TIPO" as tr_tipo, tr."TR_FECHA" as tr_fecha, tr."TR_EJERCICIO" as tr_PE_EJERCICIO, tr."TR_IMPORTE" as tr_importe, tr."TR_ESTATUS" as tr_estatus,
               tr."TR_CONCEPTO" as tr_concepto, tr."TR_DOCUMENTO_SOPORTE" as tr_documento_soporte,
               jsonb_build_object('pe_no', peo."PE_NO") AS origen, jsonb_build_object('pe_no', ped."PE_NO") AS destino
        FROM presupuesto.transferencias_ppto tr
        LEFT JOIN presupuesto.presupuesto_egresos peo ON tr."TR_PE_ORIGEN" = peo."PE_NO" AND peo."PE_EJERCICIO" = v_PE_EJERCICIO
        LEFT JOIN presupuesto.presupuesto_egresos ped ON tr."TR_PE_DESTINO" = ped."PE_NO" AND ped."PE_EJERCICIO" = v_PE_EJERCICIO
        WHERE tr."TR_EJERCICIO" = v_PE_EJERCICIO
          AND (p_id IS NULL OR tr."TR_NO" = p_id)
          AND (p_folio IS NULL OR tr."TR_FOLIO" ILIKE '%' || p_folio || '%')
          AND (p_tipo IS NULL OR tr."TR_TIPO" = p_tipo)
          AND (p_estatus IS NULL OR tr."TR_ESTATUS" = p_estatus)
        ORDER BY tr."TR_NO" DESC LIMIT GREATEST(p_limit, 1) OFFSET v_offset
    ) t;
    
    RETURN jsonb_build_object('error', 0, 'mensaje', 'OK', 'PE_EJERCICIO', v_PE_EJERCICIO, 'rowcount', COALESCE(jsonb_array_length(v_result), 0), 'data', COALESCE(v_result, '[]'::jsonb));
EXCEPTION WHEN OTHERS THEN 
    RETURN jsonb_build_object('error', 1, 'mensaje', SQLERRM, 'rowcount', 0, 'data', '[]'::jsonb);
END;
$function$

CREATE OR REPLACE FUNCTION presupuesto.fn_leer_transferencias_recalendario_meses(
    p_id bigint DEFAULT NULL,
    p_tr_no bigint DEFAULT NULL,
    p_pe_no bigint DEFAULT NULL,
    p_tipo_afectacion smallint DEFAULT NULL,
    p_estatus integer DEFAULT NULL,
    p_page integer DEFAULT 1,
    p_limit integer DEFAULT 50,
    p_pe_ejercicio_trabajo integer DEFAULT NULL,
    p_mes_trabajo smallint DEFAULT NULL,

CREATE OR REPLACE FUNCTION presupuesto.fn_contabilizar_transferencias_ppto(p_tr_no bigint, p_usuario character varying)
 RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER AS $function$
DECLARE 
    v_estatus smallint; v_det record; v_disp record;
BEGIN
    SELECT "TR_ESTATUS" INTO v_estatus FROM presupuesto.transferencias_ppto WHERE "TR_NO" = p_tr_no;
    IF v_estatus <> 2 THEN RETURN jsonb_build_object('error', 1, 'mensaje', 'Debe estar en estatus 2.'); END IF;

    FOR v_det IN SELECT * FROM presupuesto.transferencias_recalendario_meses WHERE "TRM_TR_NO" = p_tr_no LOOP
        SELECT * INTO v_disp FROM presupuesto.presupuesto_egresos_meses WHERE "PEM_PE_NO" = v_det."TRM_PE_NO" FOR UPDATE;
        
        IF v_det."TRM_TIPO_AFECTACION" = 1 THEN
            -- BAJA: Validar sobregiro...
            IF v_disp."PEM_MES_ENE" < v_det."TRM_MES_ENE" OR v_disp."PEM_MES_FEB" < v_det."TRM_MES_FEB" THEN
                RETURN jsonb_build_object('error', 1, 'mensaje', 'Sobregiro detectado...');
            END IF;
            -- Aplicar BAJA (Update)
            UPDATE presupuesto.presupuesto_egresos_meses SET "PEM_MES_ENE" = "PEM_MES_ENE" - v_det."TRM_MES_ENE" ...
            WHERE "PEM_PE_NO" = v_det."TRM_PE_NO";
        ELSIF v_det."TRM_TIPO_AFECTACION" = 2 THEN
            -- Aplicar ALTA (Update)
            UPDATE presupuesto.presupuesto_egresos_meses SET "PEM_MES_ENE" = "PEM_MES_ENE" + v_det."TRM_MES_ENE" ...
            WHERE "PEM_PE_NO" = v_det."TRM_PE_NO";
        END IF;

        -- Actualizar Total
        UPDATE presupuesto.presupuesto_egresos_meses SET
           "PEM_CALENDARIZADO" = "PEM_MES_ENE"+"PEM_MES_FEB"+"PEM_MES_MAR"+"PEM_MES_ABR"...
        WHERE "PEM_PE_NO" = v_det."TRM_PE_NO";
    END LOOP;

    UPDATE presupuesto.transferencias_ppto SET "TR_ESTATUS" = 3 WHERE "TR_NO" = p_tr_no;
    RETURN jsonb_build_object('error', 0, 'mensaje', 'Transferencia contabilizada.');
EXCEPTION WHEN OTHERS THEN RETURN jsonb_build_object('error', 1, 'mensaje', 'Error al contabilizar: ' || SQLERRM);
END;$function$

7 Acciones del Módulo y Procedimientos

Esta sección mapea los eventos transaccionales del sistema con los pasos normativos y operativos. Cada acción puede alterar el estatus de la transferencia y detonar procesos subyacentes como la validación de saldos y la generación de pólizas contables.

Acción (Botón UI)Rol RequeridoCambio de EstatusPaso Operativo / Regla de Negocio
Guardar Transferencia (Borrador) Enlace Administrativo / Capturista Se mantiene en 1 (CAPTURA) Paso Inicial: Se registran los datos de cabecera, el oficio soporte y la distribución de recalendarización (origen/destino). No afecta saldos reales aún.
Enviar a Autorización Enlace Administrativo De CAPTURA a 2 (EN REVISIÓN) El usuario de la Unidad Administrativa remite el trámite para su escrutinio por la Dirección de Presupuesto. A partir de aquí no puede editar.
Autorizar (y Generar Póliza) Director / Control Presupuestal A 2 (AUTORIZADO) Validación Normativa: Se verifica liquidez de la partida origen. Se crea automáticamente la póliza contable que asienta el movimiento de transferencia, referenciada en TR_POLIZA_NO.
Contabilizar (Afectación Física) Sistema / Control Presupuestal De AUTORIZADO a 3 (CONTABILIZADA) Paso Crítico: Ejecuta fn_contabilizar_transferencias_ppto para descontar físicamente los recursos del calendario de la partida origen e incrementarlos en la destino.
Rechazar / Cancelar Autorizador / Capturista A 4 (CANCELADA) El trámite queda nulo. Si la transferencia aún estaba en captura, el creador puede darla de baja lógica. Si estaba en revisión y hay un error, el autorizador la rechaza (Cancela).

9 Reglas de Negocio Críticas (Normatividad CONAC y LGCG)
Particionamiento por Ejercicio Fiscal: A partir de 2024, para garantizar un desempeño óptimo y mantener la segregación de la contabilidad gubernamental, las tablas transferencias_ppto y transferencias_recalendario_meses están particionadas nativamente por la columna EJERCICIO mediante PARTITION BY LIST en PostgreSQL.
CódigoRegla / Justificación NormativaConsecuencia / Acción del SistemaDónde se valida
RN-TRA-01Suma Cero obligatoria: La suma de la calendarización (Ene-Dic) debe cuadrar exactamente con el Importe Total.No se permite guardar si hay diferencias entre el Origen y el Destino.Frontend / SP

RN-TRA-03 Prohibición Inter-Capítulos (Art. 9 LGCG):
El Procedimiento 11 exige apego a clasificadores CONAC. Las transferencias entre distintos capítulos de gasto (Ej. de Materiales 2000 a Servicios 3000) violan la disciplina financiera.		 El Frontend lee el 1er dígito de la partida (substring 15). Si se mezcla, lanza alerta roja y bloquea la adición al carrito. Frontend (UI)
RN-TRA-04 Validación de Sesión Activa (Art. 37-38 LGCG):
Los registros contables deben reflejarse en tiempo real. No se permite capturar transferencias con fecha de ejercicios dispares.		 Si el año/mes del input de fecha no coincide con el ejercicio/mes activo de la sesión del usuario, se bloquea el guardado. Frontend (UI)
RN-TRA-05 Bloqueo de Meses Cerrados:
Paso 23 del Procedimiento Interno dicta rechazar afectaciones a meses cerrados.		 En el modal de captura, los inputs de los meses anteriores al mes activo de la sesión estarán disabled (bloqueados gráficamente). Frontend (UI)
RN-TRA-06Protección de Piso Devengado: Ninguna reducción puede superar el "Disponible" real de la cuenta origen en el mes afectado.Bloqueo transaccional.Frontend / SP
RN-TRA-07 Asignación de Folio:
El TR_FOLIO es de solo lectura y se genera automáticamente al guardar la cabecera. Si es nueva transferencia, el campo se muestra vacío.		 Se visualiza un bloque de "Folio" en modo disabled en la pantalla de captura. Frontend / SP
RN-TRA-08 Fecha Sugerida Inteligente (Días Hábiles):
  • Si Mes Activo == Mes Actual: Hoy.
  • Si Mes Activo < Mes Actual: Último día hábil (Lun-Vie) del Mes Activo.
  • Si Mes Activo > Mes Actual: Primer día (01) del Mes Activo.
 Al abrir el modal, la fecha se pre-calcula e inyecta al campo automáticamente basado en el ejercicio y mes de la sesión del usuario. Frontend (UI)
RN-TRA-09 Búsqueda Dinámica de Partidas:
El catálogo de partidas a transferir se filtra consumiendo la función presupuesto.fn_lista_partidas_por_clave, pasando como parámetros Organización, Función, Programa, Subprograma y Proyecto. El Capítulo 1000 está excluido de las transferencias.		 Modal (Alta / Baja): Los dropdowns de la estructura programática actúan en cascada para poblar el input de búsqueda predictiva. Frontend / SP
RN-TRA-10 Acción - Actualizar Totales:
El botón "Actualizar Totales" dentro del modal recalcula en tiempo real las sumatorias de Total Bajas, Total Altas y la Diferencia, basándose en los montos capturados en la cuadrícula de los 12 meses.		 Modal (Calendario Mensual) Frontend (UI)

Al presionar "Calcular Diferencia", el sistema infiere el monto faltante para cuadrar el movimiento a cero (Total Altas = Total Bajas) y lo inyecta automáticamente en el input del mes correspondiente a la Fecha de Operación activa.		 Modal (Calendario Mensual) Frontend (UI)
