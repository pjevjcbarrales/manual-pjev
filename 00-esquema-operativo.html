Metodología y Esquema Operativo

Herramientas, reglas de colaboración, manejo de ramas, rutinas diarias y despliegue del proyecto SIAF-PJEV.

1 Entorno de Desarrollo y Stack Tecnológico
ℹ
El objetivo es mantener el flujo lo más simple posible: Equipos locales de desarrollo, 1 Base de Datos en Nube, 1 Servidor de Pruebas.

El entorno técnico para construir el sistema opera sobre sistemas operativos Windows / Unix / Linux y el código se administra mediante el repositorio GitHub con una configuración de ramas básica. La herramienta oficial de desarrollo es el IDE Visual Studio Code. Todo el ecosistema está basado en el lenguaje JavaScript/TypeScript.

CapaTecnologías Oficiales
Base / LenguajeTypeScript, HTML, CSS
FrontendNext.js, Tailwind.UI
Backend (Servicios REST)Node.js, Nest.js
PersistenciaPostgreSQL, Prisma.js
Generación de ReportesReact/JS (Reportes Simples), JasperReports (Reportes Complejos)

3 Roles y Responsabilidades

El equipo se organiza en base a los siguientes roles funcionales. Una sola persona puede asumir uno o múltiples roles:

Análisis y Diseño Funcional
  • Levantamiento de requerimientos y reglas de negocio.
  • Elaboración de manuales de desarrollo.
  • Actualizar este sitio web de documentación.
Administrador de Base de Datos
  • Gestión de la estructura en la base de datos centralizada.
  • Asegurar integridad, esquemas y aplicar migraciones SQL.
Desarrollo de Aplicación
  • Programación en las capas de frontend, backend y persistencia.
  • Trabajo diario en su rama individual dev-usuario.
Revisión e Integración
  • Revisión de Pull Requests (Code Review y QA).
  • Responsable autorizado de hacer los merge hacia las ramas de integración (ej. development y producción).
Actualización en Producción
  • Responsable de desplegar el código aprobado hacia los servidores finales.
4 Flujo de Trabajo Diario (Rutinas en GitHub)

La forma de trabajo principal consiste en que cada desarrollador tiene su rama de trabajo individual (dev-usuario) para actualizar el repositorio. Para evitar conflictos, se debe cumplir con las siguientes rutinas de control de versiones:

🌅 Rutina de Mañana (09:00 hrs) — Sincronización

# 1. Traer el mapa remoto actualizado
git fetch origin

# 2. Actualizar tu copia local de la rama principal
git checkout development
git pull origin development

# 3. Llevar las nuevas actualizaciones a tu rama personal
git checkout dev-usuario
git merge development

# 4. Refrescar el monorepo técnico
npm install
npx prisma generate --schema=packages/database/prisma/schema.prisma

🌇 Rutina de Tarde — Respaldo y Entrega

# 1. Preparar archivos modificados
git add .

# 2. Documentar avance (usando el estándar semántico feat: o fix:)
git commit -m "feat: (modulo) descripción clara del avance hoy"

# 3. Subir los cambios a la nube en tu rama
git push origin dev-usuario
5 ¿Cuándo está listo un cambio?

Según la metodología oficial del equipo, todo cambio o módulo nuevo debe cumplir con el siguiente Checklist de Terminado (Definition of Done):

EstadoCriterio a cumplir rigurosamente
Requerido Todo el código es TypeScript. Está estrictamente prohibido utilizar o enviar archivos .js.
Requerido Bitácora de BD: Los cambios en la base de datos (tablas, funciones) están trazados mediante un script .sql. No se modifican tablas directamente en el editor de Supabase sin actualizar el esquema de Prisma.
Requerido Se ha generado el Pull Request (PR) en GitHub solicitando fusionar dev-usuario con la rama principal.
Requerido El responsable de "Revisión" ha aprobado el PR y ha ejecutado la fusión (merge) hacia development o producción.
Requerido El responsable de "Actualización en producción" ha desplegado los cambios y la funcionalidad se probó exitosamente.
