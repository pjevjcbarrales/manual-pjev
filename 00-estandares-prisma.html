Estándar de Prisma ORM

Lineamientos oficiales para el modelado, mapeo de esquemas y conexión con la base de datos utilizando Prisma como nuestro ORM en Nest.js.

1 Prisma en la Arquitectura

En nuestra arquitectura, Prisma ORM no se encarga de definir migraciones estructurales complejas (como el particionamiento de tablas), sino que funge como el puente tipado entre la base de datos PostgreSQL y nuestra API en Nest.js.

  • Lectura y Tipado: Todos los modelos en Prisma existen para que TypeScript conozca la estructura de los catálogos y tablas operativas.
  • Ejecución de Lógica: Debido a que nuestra lógica de negocio reside en PostgreSQL (vía Stored Procedures), Prisma se utiliza intensivamente a través de $queryRaw para ejecutar las funciones fn_leer y fn_abc.
2 Reglas para el archivo schema.prisma

Para respetar los estándares de base de datos definidos en nuestra arquitectura técnica, la declaración de los modelos en Prisma debe apegarse estrictamente a las siguientes reglas:

⚠
Soporte Multi-esquema: Dado que PostgreSQL divide la base de datos en 9 esquemas (nav, config, contabilidad, etc.), el archivo schema.prisma debe tener habilitado el feature multiSchema y cada modelo debe especificar a qué esquema pertenece usando @@schema("nombre").

Convenciones de Nomenclatura en Prisma

  • Nombres de Modelos (PascalCase): Los modelos se declaran en singular y PascalCase (ej. CatalogoProveedores, PolizaMovto).
  • Mapeo de Tablas (snake_case plural): Se debe usar la directiva @@map("nombre_tabla") para vincular el modelo con el nombre real físico de la tabla en PostgreSQL.
  • Campos idénticos a SQL: Los nombres de los campos en Prisma deben conservar la nomenclatura de la base de datos (con su prefijo correspondiente, ej. cp_id, cp_rfc) a menos que se mapeen explícitamente con @map().
3 Ejemplo Oficial de un Modelo en Prisma
model CatalogoProveedores {
  cp_id               Int                @id @default(autoincrement())
  cp_numpro           String             @db.VarChar(5)
  cp_razonsocial      String             @db.VarChar(150)
  cp_rfc              String             @unique(map: "uq_prov_rfc") @db.VarChar(14)
  cp_banco_id         Int?
  estatus_registro    Int                @default(1) @db.SmallInt
  usuario_c           String             @db.VarChar(20)
  fecha_c             DateTime           @default(now()) @db.Timestamptz(6)
  
  // Relaciones
  banco               CatalogoBancos?    @relation(fields: [cp_banco_id], references: [ban_id])

  // Mapeo Físico
  @@map("catalogo_proveedores")
  @@schema("config")
}
4 Flujo de Trabajo (DB Pull vs Migrations)

Debido a que el DBA se encarga de crear las estructuras complejas y los Stored Procedures directamente en SQL (Supabase), los desarrolladores backend deben usar el comando prisma generate para reflejar los cambios tras descargar el código de Supabase. Prisma actúa predominantemente en modo DB First para este proyecto.
