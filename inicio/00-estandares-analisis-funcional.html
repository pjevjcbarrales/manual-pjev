Estándar de Análisis Funcional (DOC-08)

Estructura obligatoria para documentar el levantamiento de requerimientos y diseño técnico de nuevos módulos.

Tipo: Normativa Interna
1 Objetivo

El Análisis Funcional es la piedra angular del desarrollo. Define el "Qué" y el "Por qué" de un módulo antes de escribir una sola línea de código. Ningún módulo debe desarrollarse sin que su Análisis Funcional correspondiente haya sido redactado, revisado y autorizado.

Todos los archivos de Análisis Funcional deben nombrarse con la convención 0X.Y.Z-nombre-modulo.html (por ejemplo, 02.1.1-clave-presupuestal.html) y deben seguir estrictamente la estructura detallada a continuación.

2 Estructura Obligatoria de 12 Puntos

1. Descripción General

Un resumen de 1-2 párrafos que explique el propósito del módulo en un lenguaje no técnico y orientado al negocio.

2. Marco Normativo

Tabla obligatoria que vincula el módulo con la normatividad. Columnas: Norma, Artículo / Punto, Obligación que genera.

3. Roles que intervienen

Tabla de accesos y permisos. Columnas: Rol, Acción permitida, Restricción.

4. Modelo de Datos

Tabla exhaustiva con los campos físicos en la base de datos (desglosado por Maestro/Detalle si aplica), marcando llaves (PK, FK) y requeridos. Columnas: Campo, Tipo, Descripción.

5. Stored Procedures (SPs) y Triggers

Debe documentarse la firma de las funciones transaccionales (Ej. fn_leer, fn_crud) y detallar reglas implementadas en Triggers de integridad. La presentación debe estructurarse usando pestañas navegables con showSP() que muestren las tablas de Entrada (IN), Valores Devueltos (OUT) en JSON y la Definición SQL embebida en un bloque formativo.

6. Servicios REST

Endpoints que el Frontend consumirá. Utilizar las etiquetas CSS .method. Columnas: Método (GET/POST/PATCH/DELETE), Endpoint, Body / Params, Descripción.

7. Acciones del Módulo

Secciones del front, tales como: Registro (CRUD), Especiales (Importaciones, Reportes) o Acciones en Vacío.

8. Diseño de Interfaces (UI / Wireframes)

Wireframes (dibujados en SVG embebido) organizados mediante pestañas interactivas manejadas por showWF(). Debe contener vistas de Grid principal, formularios modales o Steppers de validación, ilustrando estados abstractos (parámetros) y representaciones con valores de negocio concretos.

9. Reglas de Negocio Críticas

El corazón de la lógica funcional. Columnas estrictas: Código (ej. RN-01), Regla, Consecuencia, Dónde se valida.

10. Flujos de Integración (Impacto Transversal)

Explicación clara de cómo las acciones de este módulo detonan efectos en otras áreas del ERP (Ej. Impacto en presupuesto y contabilidad simultáneamente).

11. Afectación Contable (Reglas CONAC)

Describe si la operación genera o no pólizas automáticas, definiendo los momentos contables que impacta y la matriz de conversión implicada. Columnas: Aplica, Descripción Contable.

12. Historial de Cambios

Control de versiones del documento. Columnas: Fecha, Versión, Descripción del Cambio, Autor.

3 Anexos Obligatorios (Diseño para Impresión)

Adicional a las 12 secciones de vista en pantalla, el estándar exige la inclusión de 2 anexos ocultos que solo se visualizarán (y añadirán saltos de página) al imprimir el documento o exportarlo a PDF mediante la regla @media print definida en el shared.css.

Anexo 1: Diagrama Interfuncional del Procedimiento

Un contenedor <div id="module-flowchart" style="display:none;"> que alberga un diagrama SVG detallando el flujo del proceso cruzado por carriles (Área Usuaria vs SIAF-PJEV).

Anexo 2: Flujograma de Procesos y Descripción

Un contenedor <div id="module-process-table" style="display:none;"> que contenga una tabla narrativa (.props-table) detallando cada paso del diagrama interfuncional.

4 Funciones JavaScript del Estándar

Para asegurar la interactividad consistente de los Wireframes y las firmas de Stored Procedures sin cargar librerías externas excesivas, cada documento debe incluir (o referenciar globalmente en shared.js) estas dos funciones transicionales:

// Intercambiador de pestañas para Wireframes UI (Sección 8)
function showWF(id, btn) {
  document.querySelectorAll('.wf-panel').forEach(p => p.classList.remove('active'));
  document.querySelectorAll('.wf-tab').forEach(b => b.classList.remove('active'));
  document.getElementById('wf-' + id).classList.add('active');
  btn.classList.add('active');
}

// Intercambiador de pestañas para Stored Procedures (Sección 5)
function showSP(id, btn) {
  document.querySelectorAll('.sp-panel').forEach(p => p.classList.remove('active'));
  document.querySelectorAll('.sp-tab').forEach(b => b.classList.remove('active'));
  document.getElementById('sp-' + id).classList.add('active');
  btn.classList.add('active');
}
