Estándar de Servicios REST

Mapeo oficial de rutas HTTP en el backend (Nest.js / Node.js) y su vinculación con la lógica de negocio en la base de datos.

1 Responsabilidad de la Capa REST (Nest.js)

Debido a que toda la lógica de validación, joins y reglas de negocio reside en los Stored Procedures de PostgreSQL, la API en Nest.js funciona como un puente limpio (proxy). Su responsabilidad principal es:

  • Manejar la autenticación (JWT) y autorización de la ruta.
  • Mapear los verbos HTTP (GET, POST, PUT, DELETE) hacia la función correspondiente en la base de datos ejecutando Prisma $queryRaw.
  • Traducir el cod_resultado de la base de datos hacia el Código de Estado HTTP (HTTP Status Code) correcto.
2 Mapeo de Rutas para Formularios (Catálogos)

Todo controlador asociado a un catálogo o formulario simple debe exponer estrictamente las siguientes rutas, mapeándolas a los procedimientos fn_leer y fn_abc:

GET /api/{modulo} fn_leer_*(p_id = NULL)
GET /api/{modulo}/nuevo fn_leer_*(p_id = -1)
GET /api/{modulo}/:id fn_leer_*(p_id = :id)
POST /api/{modulo} fn_abc_*(accion = 1)
PUT /api/{modulo}/:id fn_abc_*(accion = 2)
DELETE /api/{modulo}/:id fn_abc_*(accion = 3)
3 Traducción de Códigos a HTTP Status

El backend debe leer la propiedad cod_resultado devuelta por la base de datos y responder al frontend (Next.js) con el código HTTP correspondiente:

Devuelto por SPSignificadoHTTP Status (Nest.js)
0 Éxito / Operación completada 200 OK o 201 Created
-1 Warning / Violación a Regla de Negocio 400 Bad Request o 422 Unprocessable Entity
-2 Danger / Error Técnico o Excepción 500 Internal Server Error
4 Estructura de Archivos del Módulo

Dentro del directorio del backend en el monorepo, cada servicio se maneja con un estándar rígido de separación de responsabilidades. La carpeta de cada módulo debe contener estrictamente la siguiente estructura:

Archivo / DirectorioResponsabilidad y Configuración
{modulo}.module.ts Orquestador del dominio. Aquí se deben registrar (importar) el controlador y los proveedores (services). Se asegura de que el módulo sea empaquetable y reutilizable.
{modulo}.controller.ts Capa de red HTTP. Mapea las rutas (usando @Get, @Post, etc.), invoca los métodos del Service y retorna la respuesta procesada. Aquí van todos los decoradores de Swagger.
{modulo}.service.ts Capa de lógica (Puente). Contiene la inyección del PrismaService. Aquí es donde se construye y dispara el $queryRaw llamando a los Stored Procedures de la base de datos (fn_leer, fn_abc).
/dto/ Carpeta obligatoria que contiene los Data Transfer Objects (DTOs) para validar estrictamente la forma del payload JSON que envía el frontend. Debe contener al menos dos archivos:
crear-{modulo}.dto.ts Clase con decoradores de validación (class-validator) para la creación (Acción 1). Documenta los campos obligatorios usando @ApiProperty() de Swagger.
actualizar-{modulo}.dto.ts Clase para actualizaciones (Acción 2). Típicamente hereda del DTO de creación usando PartialType() de Nest.js, volviendo todos los campos opcionales pero manteniendo las validaciones.
5 Ejemplos de Código (Nest.js)

Para asegurar que todos programen igual, aquí tienes la plantilla base dividida en sus 4 archivos esenciales:

1. proveedores.module.ts

import { Module } from '@nestjs/common';
import { ProveedoresController } from './proveedores.controller';
import { ProveedoresService } from './proveedores.service';

@Module({
  controllers: [ProveedoresController],
  providers: [ProveedoresService],
})
export class ProveedoresModule {}

2. dto/crear-proveedor.dto.ts

import { ApiProperty } from '@nestjs/swagger';
import { IsString, IsNotEmpty } from 'class-validator';

export class CrearProveedorDto {
  @ApiProperty({ example: 'XAXX010101000' })
  @IsString()
  @IsNotEmpty()
  rfc: string;

  @ApiProperty({ example: 'Empresa S.A.' })
  @IsString()
  @IsNotEmpty()
  razonsocial: string;
}

3. proveedores.service.ts

import { Injectable, BadRequestException, InternalServerErrorException } from '@nestjs/common';
import { PrismaService } from '../prisma.service';

@Injectable()
export class ProveedoresService {
  constructor(private readonly prisma: PrismaService) {}

  async leer(id: number | null, filtro?: string) {
    const res = await this.prisma.$queryRaw`SELECT config.fn_leer_proveedores(${id}, ${filtro}) AS payload`;
    return this.procesarRespuesta(res);
  }

  async abc(accion: number, dto: any) {
    const res = await this.prisma.$queryRaw`SELECT config.fn_abc_proveedores(${accion}, ${dto}) AS payload`;
    return this.procesarRespuesta(res);
  }

  private procesarRespuesta(resultadoSql: any) {
    const json = resultadoSql[0].payload;
    if (json.cod_resultado === -1) throw new BadRequestException(json);
    if (json.cod_resultado === -2) throw new InternalServerErrorException(json);
    return json;
  }
}

4. proveedores.controller.ts

import { Controller, Get, Post, Body, Param, Query } from '@nestjs/common';
import { ApiTags, ApiOperation } from '@nestjs/swagger';
import { ProveedoresService } from './proveedores.service';
import { CrearProveedorDto } from './dto/crear-proveedor.dto';

@ApiTags('Proveedores')
@Controller('api/catalogos/proveedores')
export class ProveedoresController {
  constructor(private readonly service: ProveedoresService) {}

  @Get()
  @ApiOperation({ summary: 'Listado general' })
  findAll(@Query('filtro') filtro: string) {
    return this.service.leer(null, filtro);
  }

  @Get('nuevo')
  getNew() {
    return this.service.leer(-1);
  }

  @Get(':id')
  findOne(@Param('id') id: number) {
    return this.service.leer(id);
  }

  @Post()
  create(@Body() dto: CrearProveedorDto) {
    return this.service.abc(1, dto);
  }
}
6 Pruebas y Documentación (Swagger)
⚡
Todos los servicios REST desarrollados en Nest.js deben documentarse y probarse rigurosamente utilizando Swagger (OpenAPI) antes de ser consumidos por el Frontend.

Swagger funciona como nuestra herramienta oficial de pruebas y como la documentación viva de nuestra API. Para asegurar el estándar de calidad, cada desarrollador backend debe cumplir con lo siguiente:

  • Decoradores Obligatorios: Todos los Controllers y DTOs en Nest.js deben estar decorados utilizando @nestjs/swagger (ej. @ApiTags, @ApiOperation, @ApiResponse).
  • Pruebas de Integración: Las acciones de Alta, Baja, Cambio y Lectura deben ejecutarse directamente en la interfaz de Swagger local (típicamente disponible en http://localhost:3000/api/docs).
  • Validación de Payload: El desarrollador debe verificar visualmente en Swagger que la respuesta del servidor empate exactamente con el JSON estándar (códigos 0, -1 y -2) devuelto por el Stored Procedure.
