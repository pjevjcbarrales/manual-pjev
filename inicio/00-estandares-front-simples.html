Estándar Frontend: Formularios Simples

Arquitectura oficial para la construcción de interfaces de catálogos. Define la relación estricta entre las vistas de Next.js y los procedimientos almacenados (Leer y ABC).

1 La Anatomía de un Catálogo (2 Fases)

Todos los módulos catalográficos del SIAF-PJEV deben estructurarse de manera uniforme utilizando un flujo de dos fases, donde la Interfaz de Usuario se acopla directamente a los procedimientos almacenados de base de datos.

Fase 1: Grid de Datos (Procedimiento LEER)

La pantalla inicial del módulo tiene como propósito presentar el universo de registros. Esta vista siempre debe incluir:

  • Filtros Superiores: Un buscador por texto o parámetros relevantes (ej. buscar por nombre o RFC).
  • Botones de Acción Global: En la parte superior derecha se sitúa el botón principal + Nuevo registro. A su vez, se deben incluir opciones para Imprimir resultados, Exportar y (cuando aplique al módulo) Importar información.
  • Carga de Información: Al renderizar la tabla o aplicar filtros, el Frontend invoca al procedimiento SP_[Modulo]_Leer pasando los parámetros de búsqueda. Lo que devuelve este SP es lo que se imprime directamente en el Grid.
Fase 2: Ventana de Registro y Actualización (Procedimientos LEER y ABC)

Al hacer click en "Nuevo registro" o "Editar", se debe mostrar una ventana dedicada donde el usuario captura la información de manera ordenada.

  • Agrupación Visual: El formulario nunca debe ser una lista plana interminable de inputs. Se deben crear "tarjetas" o bloques semánticos (ej. "Datos Personales", "Documentos", "Contacto") para facilitar la lectura.
  • Inicialización (El Procedimiento LEER): Cuando la ventana de "Nuevo Registro" se abre, el sistema ejecuta el procedimiento SP_[Modulo]_Leer (enviando el modo 'NUEVO' o pasando un ID nulo) para que la base de datos devuelva la estructura vacía, valores por defecto, o listas desplegables necesarias para pintar el formulario.
  • Guardado (El Procedimiento ABC): Cuando el usuario presiona "Guardar", toda la información recabada en el formulario se envía al backend, el cual invoca SP_[Modulo]_ABC.

    Si es un registro nuevo, se envía con la operación 'I' (Insert). Si es edición, se envía con 'U' (Update).

3 Traducción a Código (Consumo de la API REST)

A nivel de React (Frontend), este flujo se traduce en invocaciones de estado y llamadas a los endpoints REST que conectan a la base de datos.

// ----------------------------------------------------
// FASE 1: GRID (Inicialización y Búsqueda)
// ----------------------------------------------------
const cargarGrid = async (filtros) => {
  // Llama internamente al SP_Modulo_Leer
  const res = await fetch(`/api/modulo/leer?busqueda=${filtros}`);
  const datos = await res.json();
  setFilasDelGrid(datos);
};

// ----------------------------------------------------
// FASE 2: VENTANA DE REGISTRO (Inicializar Nuevo)
// ----------------------------------------------------
const abrirNuevoRegistro = async () => {
  // Llama internamente al SP_Modulo_Leer en modo inicialización (ID=null)
  const res = await fetch(`/api/modulo/leer/nuevo`);
  const defaults = await res.json();
  setCatalogosDesplegables(defaults.catalogos);
  abrirVentana();
};

// ----------------------------------------------------
// FASE 2: VENTANA DE REGISTRO (Al hacer click en Guardar)
// ----------------------------------------------------
const guardarRegistro = async (formData) => {
  const operacion = esEdicion ? 'PUT' : 'POST';
  
  // Llama internamente al SP_Modulo_ABC ('U' o 'I')
  const res = await fetch(`/api/modulo/abc`, {
    method: operacion,
    body: JSON.stringify(formData)
  });
  
  // El frontend lee la respuesta de base de datos para mostrar la notificación.
  mostrarNotificacion(res);
};
4 Reglas de Notificación (Toasts) de la Base de Datos

En el último paso del código anterior (mostrarNotificacion), el Frontend es responsable de interpretar los Códigos HTTP devueltos por el Procedimiento ABC y convertirlos en notificaciones visuales:

Respuesta HTTPCódigo devuelto por SP_ABCAcción Visual Requerida
200 OK / 201 Created 0 (Éxito) Notificación Verde "Registro guardado exitosamente." Cierra la ventana y regresa al Grid principal, actualizando la tabla.
400 Bad Request -1 (Regla de negocio) Notificación Amarilla Muestra el msg_resultado exacto devuelto por la Base de Datos (Ej. "El RFC ya existe"). No cierra el formulario para que el usuario pueda corregir el error.
500 Internal Error -2 (Excepción de SQL) Notificación Roja Muestra un mensaje técnico de falla.
