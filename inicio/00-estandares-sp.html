Estándar de la Lógica de Negocios

Reglas oficiales para estructurar las funciones y procedimientos almacenados (Stored Procedures) en la base de datos PostgreSQL, abarcando formularios simples y compuestos.

1 Arquitectura de Formularios

Toda la lógica de negocio, validaciones y permisos residen directamente en la base de datos a través de funciones (Stored Procedures). Para estandarizar el desarrollo, dividimos las pantallas en dos grandes categorías:

Formularios Simples (Catálogos)

Tablas maestras y catálogos de un solo nivel. Operan usando exclusivamente un par de funciones:

  • fn_leer_catalogo (Lectura)
  • fn_abc_catalogo (Escritura)
Formularios Compuestos (Maestro-Detalle)

Módulos operativos que usan un encabezado y sus líneas de movimientos (ej. Pólizas). Requieren 4 funciones:

  • fn_leer_encabezado y fn_abc_encabezado
  • fn_leer_movtos y fn_abc_movtos
2 Reglas para Funciones LEER

Las funciones de lectura fn_leer_* son el motor de consulta. Se controlan enviando un parámetro clave llamado @id y otros filtros de búsqueda opcionales. El comportamiento cambia drásticamente según el valor de @id:

Comportamiento según @id
Valor de @idAcción que realiza el SP
Nulo (IS NULL) Listado: Ejecuta un SELECT aplicando los filtros enviados por parámetro para devolver el listado general de la tabla.
Mayor a Cero (> 0) Detalle: Busca y devuelve un único registro existente filtrando por ese ID exacto. Ideal para editar o consultar a fondo.
Menor a Cero (< 0) Objeto Nuevo: Devuelve un SELECT con valores por default. El Frontend lo utiliza para inicializar un formulario en blanco listo para crear un nuevo registro.

Responsabilidades adicionales del LEER:

  • Unión de Datos (JOINs): Es aquí donde vinculamos tablas auxiliares para hacer las descripciones legibles. Ejemplo: Si la tabla trae idunidadadmva, el SP debe hacer un JOIN con config.catalogo_ua para devolver adicionalmente la claveua y la descripcion, evitando que el Frontend haga esa lógica.
  • Permisos de Consulta: En esta función se deben aplicar los filtros restrictivos dependiendo del rol del usuario (ej. si el usuario solo puede ver los registros de su dependencia).
3 Reglas para Funciones ABC (Escritura)

Las funciones fn_abc_* manejan todo el CRUD excepto la lectura. Se rigen enviando todos los datos del formulario más un parámetro vital que indica la acción a realizar:

Operaciones de Escritura
  • Acción 1 (Alta): Inserta un nuevo registro en la tabla tras validar las reglas de negocio y campos obligatorios.
  • Acción 2 (Cambio): Hace un UPDATE del registro. Modifica únicamente los campos permitidos y actualiza los campos de auditoría (usuario_m, fecha_m). Requiere el @id.
  • Acción 3 (Baja Lógica): No se utiliza DELETE físico. Hace un UPDATE estableciendo estatus_registro = 0 para conservar la auditoría.

Nota: En algunos módulos legados por homologar (según el doc GUIA.md), los valores mapean a Acción 2=Alta, 3=Cambio, 4=Baja. El estándar unificado actual debe alinearse con 1=Alta, 2=Cambio, 3=Baja para el desarrollo de nuevos módulos.

4 Envelope de Respuesta (JSON)

Tanto las funciones de LEER como de ABC deben devolver invariablemente un solo objeto JSON con un contrato unificado, para que el Frontend y la API en Nest.js puedan leerlo homogéneamente.

Código devueltoNivel de ErrorSignificado
0 Éxito (No hay error) Operación completada satisfactoriamente. Se anexa la descripción ("Registro guardado" o "OK") y la información en el nodo data.
-1 Warning (Regla de negocio) Advertencia o violación a una regla de negocio capturada (Ej. "El RFC ya existe" o "Presupuesto insuficiente").
-2 Danger (Error técnico) Excepción controlada del motor de base de datos (Ej. "Fallo en constraint" o error WHEN OTHERS).

Estructura JSON Esperada

{
  "cod_resultado": 0,
  "msg_resultado": "Descripción de éxito o error",
  "data": {
    // Para ABC: "id": 123
    // Para LEER: "registros": [...], "meta": {"total": 5}
  }
}
5 Ejemplo de Implementación (PostgreSQL)

A continuación se muestra un esqueleto estructural de cómo debe programarse la función fn_leer aplicando la lógica del parámetro p_id y manejando el JSON de respuesta:

CREATE OR REPLACE FUNCTION config.fn_leer_ejemplo(
    p_id INT,
    p_filtro VARCHAR DEFAULT NULL
)
RETURNS jsonb
LANGUAGE plpgsql
AS $$
DECLARE
    v_resultado jsonb;
BEGIN
    -- 1. OBJETO NUEVO (p_id < 0)
    IF p_id < 0 THEN
        SELECT jsonb_build_object(
            'id', 0,
            'descripcion', '',
            'activo', 1
        ) INTO v_resultado;

    -- 2. DETALLE DE REGISTRO (p_id > 0)
    ELSIF p_id > 0 THEN
        SELECT jsonb_build_object(
            'id', e.id,
            'descripcion', e.descripcion,
            'unidad_nombre', u.nombre -- JOIN auxiliar
        ) INTO v_resultado
        FROM tabla_ejemplo e
        LEFT JOIN catalogo_ua u ON e.idunidadadmva = u.id
        WHERE e.id = p_id;

    -- 3. LISTADO GENERAL (p_id IS NULL)
    ELSE
        SELECT jsonb_build_object(
            'registros', COALESCE(jsonb_agg(row_to_json(t)), '[]'::jsonb)
        ) INTO v_resultado
        FROM (
            SELECT id, descripcion 
            FROM tabla_ejemplo
            WHERE (p_filtro IS NULL OR descripcion ILIKE '%' || p_filtro || '%')
            AND estatus_registro = 1
        ) t;
    END IF;

    -- ENVELOPE DE ÉXITO
    RETURN jsonb_build_object(
        'cod_resultado', 0,
        'msg_resultado', 'OK',
        'data', v_resultado
    );

EXCEPTION WHEN OTHERS THEN
    -- ENVELOPE DE ERROR TÉCNICO
    RETURN jsonb_build_object(
        'cod_resultado', -2,
        'msg_resultado', SQLERRM,
        'data', NULL
    );
END;
$$;
