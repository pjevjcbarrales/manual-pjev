Estándar Frontend: Formularios Compuestos

Arquitectura para flujos de Maestro/Detalle (Ej. Pólizas, Órdenes de Compra). Define el ciclo de vida donde el Encabezado debe guardarse antes de poder capturar sus Partidas.

1 La Anatomía de un Compuesto (3 Fases)

A diferencia de los catálogos simples, los formularios compuestos manejan relaciones complejas de uno a muchos (1:N). Para garantizar la integridad referencial en la base de datos, el flujo de trabajo obliga a crear el Maestro primero antes de permitir la captura de Detalles.

Fase 1: Grid de Registros (Procedimiento Maestro_LEER)

Al igual que en los catálogos simples, la pantalla principal es una tabla con un buscador superior compuesto (multi-filtro: Tipo, Folio, Fechas, Estatus). Se alimenta del procedimiento de lectura del encabezado. Cuenta con un botón principal + Nuevo registro.

Fase 2: Alta del Encabezado (Procedimiento Maestro_ABC)

Pantalla inicial de captura. Se muestran únicamente los campos del maestro (Ej. Tipo de póliza, Fecha, Centro de costos, Descripción).

  • Restricción: En esta fase NO se muestra la tabla de detalles ni movimientos. Se debe mostrar un mensaje claro al usuario: "Primero capture el encabezado y cree el registro. Después podrá agregar movimientos en esta misma pantalla."
  • Acción: Al darle a "Crear", se ejecuta el procedimiento ABC del maestro. Si es exitoso, el frontend obtiene el ID o Folio recién creado y recarga la misma pantalla pasando a la Fase 3.
Fase 3: Edición y Captura de Partidas (Procedimiento Detalle_ABC)

La pantalla ahora está en "Modo Edición". Además del formulario del encabezado (cuyos cambios se guardan con Guardar encabezado), se habilitan dos nuevas zonas:

  • Barra de Acciones de Flujo: Botones para cambiar el estado del registro (Imprimir PDF, Enviar solicitud, Regresar borrador, Historial).
  • Sección de Movimientos / Partidas: Un grid interno que muestra los detalles asociados al maestro.
  • Modal de Nuevo Movimiento: Para agregar una partida, se abre un Modal flotante por encima de la pantalla. Este modal se comunica con su propio procedimiento Detalle_ABC y debe incluir los botones Guardar y Guardar y nuevo para facilitar la captura rápida de múltiples partidas.

3 Traducción a Código (SPs de Backend)

A nivel de React, este proceso de "Maestro primero, Detalles después" se debe traducir en llamadas API aisladas para preservar el rendimiento y transaccionalidad:

// ----------------------------------------------------
// 1. GUARDAR ENCABEZADO (Detona SP_Maestro_ABC)
// ----------------------------------------------------
const guardarEncabezado = async (formData) => {
  const res = await fetch(`/api/modulo/maestro/abc`, {
    method: 'POST',
    body: JSON.stringify(formData)
  });
  const data = await res.json();
  
  if (data.id) {
    // Redirigir a la vista de edición para desbloquear el modal de Detalles
    router.push(`/modulo/reg/${data.id}`);
  }
};

// ----------------------------------------------------
// 2. GUARDAR DETALLE (Detona SP_Detalle_ABC)
// ----------------------------------------------------
const guardarMovimiento = async (movimientoData, esGuardarYNuevo) => {
  const res = await fetch(`/api/modulo/detalle/abc`, {
    method: 'POST',
    body: JSON.stringify({ ...movimientoData, masterId })
  });
  
  if (res.ok) {
    recargarTablaDeMovimientos();
    if (!esGuardarYNuevo) {
      cerrarModal();
    } else {
      limpiarFormularioDeMovimiento();
    }
  }
};
