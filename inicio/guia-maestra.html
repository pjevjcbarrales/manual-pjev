Guía Maestra del Manual

Cómo leer, interpretar y utilizar el Manual de Desarrollo del SIAF-PJEV. Esta página explica la estructura de la documentación, las secciones de cada módulo y la nomenclatura utilizada en los wireframes.

1 Propósito del Manual

El Manual de Desarrollo del SIAF-PJEV es la fuente única de verdad para el equipo técnico. Su objetivo es unificar los criterios de desarrollo entre Backend, Frontend y Base de Datos, proporcionando una referencia clara sobre el modelo de datos, los Stored Procedures y las reglas de negocio de cada módulo del sistema.

2 Clasificación de Módulos

Cada módulo documentado en este sitio cuenta con insignias (badges) que indican su estado y tipo de complejidad.

InsigniaCriterioDescripción
Modo A Estado del código El módulo ya existe en el código fuente actual. La documentación se extrajo haciendo ingeniería inversa de las tablas y Stored Procedures en la base de datos.
Modo B Estado del código El módulo está por desarrollar. La documentación sirve como especificación funcional y técnica para que el equipo comience a programarlo.
Simple Complejidad UI Módulos que consisten únicamente en un grid de consulta y un formulario de encabezado simple (ej. Catálogos, Entidad).
Compuesto Complejidad UI Módulos que requieren una vista Maestro-Detalle (encabezado + líneas de movimiento o partidas, ej. Afectaciones de Gasto, Pólizas).
3 Estructura de un Módulo (10 Secciones)

Todas las páginas de módulos siguen estrictamente la misma plantilla de 10 secciones para garantizar homogeneidad:

#SecciónQué contiene
1Descripción GeneralResumen del objetivo del módulo y su lugar en el sistema.
2Marco NormativoArtículos de la LGCG o acuerdos del CONAC que justifican la existencia y reglas del módulo.
3Roles que intervienenQué tipos de usuarios (ej. Capturista, Autorizador, Administrador) interactúan con esta pantalla y qué permisos tienen.
4Modelo de DatosDiccionario de la tabla principal. Identifica PK, tipos de dato y si los campos son requeridos REQ u opcionales OPC.
5Stored ProceduresDetalle de los parámetros de entrada @param y la salida esperada en formato JSON, incluyendo bloques de código SQL para copiar.
6Servicios RESTMapeo de los endpoints de la API (NestJS) que exponen las funcionalidades del módulo hacia el frontend (Next.js).
7Pantallas (Wireframes)4 vistas SVG interactuables (P1/P2/P3/P4) que explican el diseño de la interfaz y su enlace con los datos.
8Reglas de NegocioTabla con las validaciones críticas (ej. "RN-01") y en qué capa deben verificarse (Frontend, Backend, o BD).
9Flujos de IntegraciónDiagramas o listas que explican cómo los datos de este módulo afectan o alimentan a otros subsistemas.
10Historial de CambiosRegistro de control de versiones de la documentación del módulo.
4 Cómo leer los Wireframes SVG

La sección 7 de cada módulo incluye un visor con 4 pestañas de wireframes interactivos. El objetivo de estas pantallas no es ser el diseño final (pixel-perfect), sino explicar el mapeo de datos entre el Frontend y el Backend.

Pantallas Técnicas (P1 y P3)

Usan variables en lugar de texto real:

  • @parametro (Azul): Representa un dato que el usuario captura o selecciona en el Frontend y que se envía como entrada al Stored Procedure (IN).
  • {campo} (Verde): Representa un dato que la base de datos devuelve (OUT) y que el Frontend debe pintar en la pantalla (ej. en un grid).
Pantallas de Ejemplo (P2 y P4)

Muestran la interfaz con datos reales "mockeados" para dar contexto visual al equipo de QA y UX.

  • Incluyen ejemplos de toasts de error o éxito.
  • Muestran cómo se renderiza el "envelope" JSON de respuesta real del SP.
