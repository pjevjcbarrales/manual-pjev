03.4.1.2

Libro Diario Contable

Reporte oficial exigido por el CONAC (Normativa SIAF) que detalla analíticamente los movimientos de cada cuenta contable en un periodo determinado.

Tipo: Reporte Financiero (DOC-08)
1 Descripción General

El Libro Diario es el registro contable principal donde se asientan, en orden cronológico, todas las operaciones económicas (pólizas) realizadas por la institución durante un periodo determinado.

Este reporte permite visualizar de manera resumida o detallada cada evento contable, asegurando la aplicación del principio de partida doble (Sumas Iguales) en cada póliza antes de su pase al Libro Mayor o Libro Auxiliar.

2 Marco Normativo
NormaObligación que genera
LGCG (Art. 33 y 34)La contabilidad gubernamental deberá llevarse, como mínimo, en un Libro Diario y un Libro Mayor.
CONACEmitir reportes diarios o por periodo que evidencien el registro cronológico de los asientos contables indicando cuentas, cargos y abonos.
3 Roles que intervienen
RolAcción permitida
Director de ContabilidadConsulta sin restricciones de fechas ni centros de costo. Emite el Libro Diario Oficial para la Cuenta Pública.
Analista ContableConsulta el Libro Diario para conciliaciones diarias de su unidad o centro de costo asignado.

Definición Funcional (PostgreSQL PL/pgSQL Completo)

-- ==========================================
-- Genera Libro Diario Contable
-- ==========================================
CREATE OR REPLACE FUNCTION contabilidad.sp_cont_leer_libro_diario(
    p_ejercicio INTEGER,
    p_fecha_ini DATE,
    p_fecha_fin DATE,
    p_tipos TEXT DEFAULT 'PD,PE,PI,PCH',
    p_folio_ini TEXT DEFAULT NULL,
    p_folio_fin TEXT DEFAULT NULL,
    p_centro_costo_id INTEGER DEFAULT NULL,
    p_fuente_fin_id INTEGER DEFAULT NULL,
    p_estatus TEXT DEFAULT '3',
    p_modo TEXT DEFAULT 'RESUMEN'
) RETURNS TABLE (...) AS $$
BEGIN
  -- Lógica interna:
  -- 1. Validar parámetros requeridos y convertir texto delimitado por comas a arreglos.
  -- 2. IF p_modo = 'DETALLE', cruzar cont_polizas con cont_polizas_detalles, filtrando estatus y fechas.
  -- 3. IF p_modo = 'RESUMEN', agrupar cont_polizas_detalles a nivel de póliza (sumando debe y haber).
  -- 4. Ordenar salida por pol_fecha ASC, pol_tipo ASC, pol_numero ASC.
END;
$$ LANGUAGE plpgsql;

7 Acciones del Módulo

El sistema expone las siguientes acciones operativas:

  • Filtros Complejos: El usuario puede cruzar filtros por: Rango de Fechas, Tipos de Póliza, Estatus, Folios o Centro de Costo.
  • Modo Detalle / Resumen: Permite ver la sumatoria total por póliza (resumen) o desglose por cuenta contable (detalle).
  • Alertas visuales: Informa si la consulta incluye pólizas no afectadas.

9 Reglas de Negocio Críticas
  • El reporte debe listar los movimientos cronológicamente. No calcula saldos iniciales o finales de cuentas.
  • El sistema debe mostrar una alerta si se incluyen pólizas con estatus distinto a "Afectada".
  • La seguridad por Centro de Costo u Organización restringe los movimientos que el usuario puede ver, pero siempre asegurando la advertencia visual de que la partida doble podría no cuadrar si solo se ven fracciones de una póliza.
10 Flujos de Integración (Impacto Transversal)

El Libro Diario interactúa transversalmente leyendo información de:

  • Nómina y Padrón de Proveedores: De donde surgen muchas de las pólizas automáticas.
  • Presupuesto: Para los filtros y desglose de Centro de Costo y Fuentes de Financiamiento.
11 Entregables / Reportes
  • Grid Interactivo (Pantalla): Muestra los movimientos (RESUMEN o DETALLE) directamente en la UI.
  • PDF (Formato CONAC): Reporte oficial con sellos y encabezados institucionales para archivo físico.
  • CSV / EXCEL: Archivo de datos planos para conciliación manual.
