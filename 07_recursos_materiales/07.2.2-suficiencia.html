1 Descripción General

La opción de Suficiencia Presupuestal es el primer eslabón obligatorio en la cadena de suministros del Poder Judicial. Permite al área de Recursos Materiales consultar en tiempo real el saldo de las partidas presupuestales (COG) solicitadas por las Unidades Administrativas. Su objetivo principal es asegurar que exista dinero líquido disponible antes de firmar cualquier pedido o contrato, separando los fondos de manera preventiva.

Nota Arquitectónica (Estructura Programática): El presupuesto del PJEV se clasifica estrictamente bajo la estructura: Organización, Función, Programa, Subprograma y Proyecto. Esta relación jerárquica es la base común que debe prevalecer en toda la aplicación (BD, Backend y Frontend). Ningún módulo, incluido éste, puede operar fuera de esta estructura.

Si la partida cuenta con saldo suficiente, se genera un trámite de Confirmación. Si el saldo es insuficiente en el mes actual pero existe en meses futuros, se solicita una Recalendarización. Si no existe saldo anual, se solicita una Transferencia o Apertura de partida.

2 Marco Normativo (SIAF / CONAC)
NormaObligación que genera
LGCG Art. 25 Garantizar la armonización contable-presupuestal. Ninguna adquisición o servicio puede ser comprometido sin estar respaldado previamente por disponibilidad presupuestal confirmada en la partida correspondiente.
Reglamento Interior del PJEV Obliga a la generación de un Oficio de Requerimiento firmado para que la Subdirección de Recursos Financieros avale que los recursos no exceden el Presupuesto Modificado Autorizado.
Control Interno (Auditoría) En el caso de adjudicaciones directas menores, exige la formulación y validación de una "Tarjeta DGA" interna antes de avanzar a la orden de compra.
3 Roles que intervienen
RolAcción permitidaRestricción
Analista Presupuestal (Creador) Crear la solicitud, seleccionar la partida y montos, y turnarla a firma. No puede autorizar su propio trámite ni enviarlo directamente a Recursos Financieros.
Jefe / Subdirector de Rec. Materiales Revisar y firmar digitalmente la solicitud y la Tarjeta DGA. Turnar a Recursos Financieros. Una vez turnado a Finanzas, no puede modificar los montos solicitados.
Recursos Financieros (Validador) Aprobar o Rechazar el oficio de requerimiento. Solo puede autorizar si el motor contable confirma saldo positivo en la cuenta bancaria / partida.

-- ==========================================
-- 1. LEER ENCABEZADO
-- ==========================================
CREATE OR REPLACE FUNCTION rm.fn_leer_suficiencia_encabezado(
    p_ejercicio INT,
    p_id_suficiencia INT DEFAULT NULL,
    p_id_centro_costo INT DEFAULT NULL,
    p_fecha_inicio DATE DEFAULT NULL,
    p_fecha_fin DATE DEFAULT NULL,
    p_folio_solicitud VARCHAR DEFAULT NULL,
    p_tipo_reserva INT DEFAULT NULL,
    p_estatus_reserva INT DEFAULT NULL,
    p_estatus_proceso INT DEFAULT NULL,
    p_estatus_vigencia INT DEFAULT NULL
) RETURNS jsonb LANGUAGE plpgsql AS $$
DECLARE v_resultado jsonb;
BEGIN
    SELECT COALESCE(jsonb_agg(row_to_json(t)), '[]'::jsonb) INTO v_resultado
    FROM (
        SELECT 
            m.id_suficiencia, m.folio_solicitud, m.fecha_solicitud, 
            p.folio_oficio_reserva, p.fecha_autorizacion,
            m.id_centro_costo, m.id_funcion, m.id_programa, m.id_subprograma, m.id_proyecto, m.concepto, m.monto_total, 
            m.id_contrato_origen, m.porcentaje_ampliacion, 
            m.estatus_actual
        FROM rm.rm_suficiencias m
        LEFT JOIN pr.pr_reservas p ON p.id_suficiencia = m.id_suficiencia
        WHERE m.estatus_actual >= 0
          AND (p_id_suficiencia IS NULL OR m.id_suficiencia = p_id_suficiencia)
          AND (p_id_centro_costo IS NULL OR m.id_centro_costo = p_id_centro_costo)
          AND (p_fecha_inicio IS NULL OR DATE(m.fecha_solicitud) >= p_fecha_inicio)
          AND (p_fecha_fin IS NULL OR DATE(m.fecha_solicitud) <= p_fecha_fin)
          AND (p_folio_solicitud IS NULL OR m.folio_solicitud ILIKE '%' || p_folio_solicitud || '%')
          -- Se asume que estos campos existirán en la tabla final o cruzarán con el historial
          AND (p_tipo_reserva IS NULL OR m.tipo_reserva = p_tipo_reserva)
          AND (p_estatus_reserva IS NULL OR m.estatus_actual = p_estatus_reserva)
          AND (p_estatus_proceso IS NULL OR m.estatus_proceso = p_estatus_proceso)
          AND (p_estatus_vigencia IS NULL OR m.estatus_vigencia = p_estatus_vigencia)
    ) t;
    RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'OK', 'data', v_resultado);
END;
$$;

-- ==========================================
-- 3. ABC ENCABEZADO
-- ==========================================
CREATE OR REPLACE FUNCTION rm.fn_abc_suficiencia_encabezado(
    p_accion INT,
    p_ejercicio INT,
    p_id_suficiencia INT DEFAULT NULL,
    p_id_centro_costo INT DEFAULT NULL,
    p_id_funcion INT DEFAULT NULL,
    p_id_programa INT DEFAULT NULL,
    p_id_subprograma INT DEFAULT NULL,
    p_id_proyecto INT DEFAULT NULL,
    p_concepto VARCHAR DEFAULT NULL,
    p_monto_total DECIMAL DEFAULT NULL,
    p_id_contrato_origen INT DEFAULT NULL,
    p_porcentaje_ampliacion DECIMAL DEFAULT NULL,
    p_id_tiposuficiencia INT DEFAULT NULL,
    p_id_usuario INT DEFAULT 1
) RETURNS jsonb LANGUAGE plpgsql AS $$
DECLARE 
    v_id_gen INT;
    v_folio VARCHAR;
BEGIN
    IF p_accion = 1 THEN
        -- ACCION 1: INSERTAR NUEVA SOLICITUD (FASE 1)
        v_folio := 'RM-SUF-' || to_char(now(), 'YYYY') || '-' || LPAD(CAST(nextval('rm.seq_suficiencias') AS VARCHAR), 3, '0');
        INSERT INTO rm.rm_suficiencias(ejercicio, folio_solicitud, fecha_solicitud, id_centro_costo, id_funcion, id_programa, id_subprograma, id_proyecto, concepto, monto_total, estatus_actual, id_contrato_origen, porcentaje_ampliacion, id_tiposuficiencia, fecha_registro, usuario_registro)
        VALUES(p_ejercicio, v_folio, now(), p_id_centro_costo, p_id_funcion, p_id_programa, p_id_subprograma, p_id_proyecto, p_concepto, p_monto_total, 0, p_id_contrato_origen, p_porcentaje_ampliacion, p_id_tiposuficiencia, now(), p_id_usuario)
        RETURNING id_suficiencia INTO v_id_gen;
        
        -- REGISTRAR EN BITÁCORA
        INSERT INTO rm.rm_suficiencias_historial(ejercicio, id_suficiencia, id_usuario_ejecutor, rol_accion, estatus_proceso, fecha_registro)
        VALUES(p_ejercicio, v_id_gen, p_id_usuario, 'ANALISTA_RM', 0, now());
        
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Solicitud de Suficiencia creada con éxito.', 'id_generado', v_id_gen);
    
    ELSIF p_accion = 2 THEN
        -- ACCION 2: ACTUALIZAR BORRADOR
        UPDATE rm.rm_suficiencias
        SET
            id_centro_costo = p_id_centro_costo,
            id_funcion = p_id_funcion,
            id_programa = p_id_programa,
            id_subprograma = p_id_subprograma,
            id_proyecto = p_id_proyecto,
            concepto = p_concepto,
            monto_total = p_monto_total,
            id_contrato_origen = p_id_contrato_origen,
            porcentaje_ampliacion = p_porcentaje_ampliacion,
            id_tiposuficiencia = p_id_tiposuficiencia,
            fecha_actualizacion = now(),
            usuario_actualizacion = p_id_usuario
        WHERE ejercicio = p_ejercicio AND id_suficiencia = p_id_suficiencia AND estatus_actual = 0;
        
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Solicitud actualizada correctamente.');
    END IF;
END;
$$;

-- ==========================================
-- 4. ABC PARTIDAS (REQUISICIÓN)
-- ==========================================
CREATE OR REPLACE FUNCTION rm.fn_abc_suficiencia_partidas(
    p_accion INT,
    p_ejercicio INT,
    p_id_detalle INT DEFAULT NULL,
    p_id_suficiencia INT DEFAULT NULL,
    p_cve_partida_cog VARCHAR DEFAULT NULL,
    p_descripcion_bien VARCHAR DEFAULT NULL,
    p_unidad_medida VARCHAR DEFAULT NULL,
    p_cantidad DECIMAL DEFAULT 0,
    p_precio_unitario DECIMAL DEFAULT 0,
    p_monto_total DECIMAL DEFAULT 0
) RETURNS jsonb LANGUAGE plpgsql AS $$
BEGIN
    IF p_accion = 1 THEN
        -- TODO: Ejecutar verificación previa con fn_rm_suficiencias_saldo_partida para validar el REGLA-SUF-03
        INSERT INTO rm.rm_suficiencias_partidas(ejercicio, id_suficiencia, cve_partida_cog, descripcion_bien, unidad_medida, cantidad, precio_unitario, monto_total)
        VALUES(p_ejercicio, p_id_suficiencia, p_cve_partida_cog, p_descripcion_bien, p_unidad_medida, p_cantidad, p_precio_unitario, p_monto_total);
        
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Partida de requisición agregada con éxito.');
    
    ELSIF p_accion = 2 THEN
        UPDATE rm.rm_suficiencias_partidas
        SET cve_partida_cog = p_cve_partida_cog,
            descripcion_bien = p_descripcion_bien,
            unidad_medida = p_unidad_medida,
            cantidad = p_cantidad,
            precio_unitario = p_precio_unitario,
            monto_total = p_monto_total,
            fecha_actualizacion = now()
        WHERE ejercicio = p_ejercicio AND id_detalle = p_id_detalle;
        
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Partida actualizada correctamente.');
        
    ELSIF p_accion = 3 THEN
        DELETE FROM rm.rm_suficiencias_partidas WHERE ejercicio = p_ejercicio AND id_detalle = p_id_detalle;
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Partida eliminada correctamente.');
    END IF;
END;
$$;

9 Reglas Críticas de Negocio

Importante para Desarrollo (Daryl): Las siguientes validaciones técnicas son de carácter normativo y deben implementarse estrictamente tanto en el Frontend (bloqueos visuales) como en el Backend (Stored Procedures).

  • REGLA-SUF-01 (Congruencia de Montos): El UI y el SP fn_abc_suficiencia_partidas deben asegurar matemáticamente que la suma (monto_enero + monto_febrero + ... + monto_diciembre) sea menor o igual al monto_total capturado en la cabecera (rm_suficiencias).
  • REGLA-SUF-02 (Validación Estricta de Saldo SAFPOJ): Esta es la regla crítica. El SP fn_rm_suficiencias_saldo_partida debe invocarse en 2 momentos: al Guardar la Partida y al Dar clic en Enviar. Debe hacer un SELECT a la tabla externa safpoj_saldos filtrando por p_ejercicio y cve_partida_cog. Si el monto requerido es superior al saldo_disponible, lanzar un RAISE EXCEPTION devolviendo -1 y el mensaje detallado del saldo libre para que el UI bloquee la acción.
  • REGLA-SUF-03 (Candado de Edición): La acción de Actualizar o Eliminar (Inputs habilitados y llamadas al ABC) solo es permitida si el estatus_actual en base de datos es 0 (Borrador) o 3 (Rechazado). Si estatus_actual IN (1, 2), el UI debe bloquearse (Read-Only) y los SP deben abortar cualquier instrucción.
  • REGLA-SUF-04 (Pre-condición de Envío): Para poder accionar el botón "Enviar a Finanzas", el frontend debe validar que exista por lo menos una (1) partida guardada en la grilla inferior (rm_suficiencias_partidas).
  • REGLA-SUF-05 (Integridad de Estructura Programática): Cualquier afectación o registro presupuestal (backend y frontend) debe validar que la combinación de Organización, Función, Programa, Subprograma y Proyecto sea válida y corresponda al ejercicio fiscal activo, evitando manejar estos elementos como datos aislados.
10 Flujos de Integración (Impacto Transversal)

La Solicitud de Suficiencia Presupuestal es el gatillo inicial de la cadena de suministro y detona los siguientes impactos transversales:

  • Impacto hacia Presupuesto (Finanzas): Al accionar "Enviar", el registro aparece automáticamente en la bandeja del módulo 02.2.8 de la Subdirección de Recursos Financieros. Los montos solicitados entran en un estado de "Pre-reserva" (no utilizables por otras áreas mientras se dictamina).
  • Impacto hacia Compras (Adquisiciones): Una suficiencia con estatus_actual = 2 (Autorizado) se convierte en el insumo obligatorio para el módulo 07.2.2 Registro de Pedidos/Contratos. Sin una suficiencia autorizada, el sistema bloquea cualquier intento de compromiso de compra.

ÁREA RESPONSABLE PASO / ETAPA PROCESO OPERATIVO SIAF-PJEV
Recursos Materiales 1. Captura de Solicitud Detecta la necesidad de una compra, genera una Petición de Suficiencia y captura las partidas a afectar. Carga el listado de partidas activas (COG) permitiendo seleccionar sólo las pertinentes al ejercicio fiscal.
Recursos Materiales 2. Generación y Envío de Oficio Genera el Oficio de Solicitud de Reserva Presupuestal y envía formalmente la solicitud para evaluación. Valida saldos matemáticamente contra SAFPOJ. Si existe saldo disponible, "pre-reserva" (Estatus 1). Si no hay saldo, bloquea e impide el envío.
Subdirección de Finanzas 3. Dictamen de Suficiencia Revisa la bandeja operativa (Módulo 02.2.8) y decide Rechazar o Autorizar las partidas solicitadas. Actualiza estatus. En caso de Rechazo, libera inmediatamente los saldos pre-reservados. En caso de Autorización, descuenta los saldos definitivamente.
Subdirección de Finanzas 4. Notificación de Resolución Notifica formalmente por oficio la autorización (o rechazo) de la suficiencia presupuestal a Recursos Materiales para continuar la adjudicación. Genera el Folio Automático M-XX de Reserva Contable y permite la exportación del PDF Oficial con sellos institucionales.
