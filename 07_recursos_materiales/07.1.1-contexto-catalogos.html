Tu objetivo como desarrollador Backend es utilizar estos catálogos como tablas de referencia (Llaves Foráneas / Foreign Keys) cuando crees las tablas transaccionales (operacionalización).

Catálogo de Proveedores
01.3.1 Proveedores
Función Específica
Almacenar la lista oficial de todas las empresas o personas físicas a las que el Poder Judicial les compra bienes o servicios. Contiene sus datos legales (RFC, Razón Social) y sus datos bancarios (Cuenta CLABE).
Cómo interactúa con las tablas de operacionalización
Las tablas transaccionales como rm_adjudicaciones, rm_contratos y rm_pedidos deben tener una columna llamada id_proveedor. Esta columna debe ser una llave foránea (Foreign Key) que apunte a la llave primaria (Primary Key) de la tabla de Proveedores.
Por qué interactúa
Porque es imposible registrar un Pedido o un Contrato sin saber a quién se le va a pagar. En el momento del "Devengado" (Registro de Factura), el sistema usará este enlace para decirle al departamento de Finanzas a qué cuenta bancaria debe transferir el dinero.
Clasificador Único de las Contrataciones Públicas
07.1.2 CUCOP
Función Específica
Es una lista nacional estandarizada por el gobierno que contiene todas las claves posibles de artículos y servicios que se pueden comprar (ejemplo: "Lápices", "Computadoras", "Servicio de Limpieza").
Cómo interactúa con las tablas de operacionalización
La tabla de detalles transaccionales, como rm_pedidos_detalle (donde se guardan los renglones de los artículos comprados), debe tener una columna llamada cve_cucop. Esta columna debe apuntar como llave foránea a la tabla del CUCOP.
Por qué interactúa
Porque la ley gubernamental exige que cada artículo comprado tenga una clave estandarizada. Esto permite generar reportes exactos de qué insumos se compraron en el año, sin importar si un usuario escribió "Lápiz" o "Lápices de madera".
Clasificador por Objeto del Gasto
01.2.2 COG
Función Específica
Es una lista jerárquica de partidas presupuestales que clasifica "en qué" se gasta el dinero (ejemplo: Partida 21101 = Materiales y útiles de oficina; Partida 33801 = Servicios de vigilancia).
Cómo interactúa con las tablas de operacionalización
Las tablas como rm_suficiencias (Suficiencia Presupuestal) y rm_pedidos_detalle deben tener una columna llamada cve_partida_cog que funcione como llave foránea hacia la tabla del COG.
Por qué interactúa
Porque antes de comprar algo, el sistema debe revisar si hay dinero disponible. La revisión de dinero (Suficiencia Presupuestal) no se hace por artículo, sino agrupada por la partida del COG. Si compras "Lápices" (CUCOP), el sistema resta dinero a la bolsa de "Materiales de Oficina" (COG).
Unidades Administrativas y Centros de Costos
01.2.1 / 01.2.9
Función Específica
Almacenar el organigrama del Poder Judicial (juzgados, direcciones, departamentos). Sirve para saber qué área física está solicitando la compra y qué área tiene el presupuesto asignado.
Cómo interactúa con las tablas de operacionalización
Las tablas de encabezado como rm_suficiencias y rm_pedidos deben tener una columna llamada id_centro_costo que apunte como llave foránea a estos catálogos.
Por qué interactúa
Porque cada área tiene un límite de dinero asignado. El sistema necesita saber quién originó la "Requisición" (Solicitud) para descontar el dinero exclusivamente de la bolsa (presupuesto) asignada a ese juzgado o departamento en particular, y no del presupuesto general.
Fuentes de Financiamiento
01.2.4 Fuentes Finan.
Función Específica
Identificar el origen del dinero. El dinero del gobierno puede venir de recursos federales, recursos estatales o ingresos propios.
Cómo interactúa con las tablas de operacionalización
Las tablas transaccionales de compromisos (rm_pedidos, rm_contratos) deben incluir una columna id_fuente_financiamiento como llave foránea.
Por qué interactúa
Porque el departamento de Finanzas no junta todo el dinero en una sola cuenta de banco. Al registrar un Contrato o un Pedido (momento del Comprometido), el sistema debe etiquetar con qué "tipo de dinero" se va a pagar, para que el sistema contable sepa de qué cuenta bancaria sacar el recurso.
Checklist por Tipo de Contratación
07.1.3 Checklist
Función Específica
Es una lista de requisitos y documentos obligatorios que el usuario debe cumplir dependiendo de si la compra es por "Adjudicación Directa", "Invitación a 3 personas" o "Licitación".
Cómo interactúa con las tablas de operacionalización
La tabla rm_compras o la tabla de expediente debe cruzarse con este catálogo para generar dinámicamente casillas de verificación (checkboxes) en la pantalla.
Por qué interactúa
Para evitar errores humanos. El sistema usará este catálogo para impedir que el usuario avance al "Registro de Contrato" si no ha verificado físicamente que entregó todos los documentos requeridos por la ley para ese tipo de compra.
