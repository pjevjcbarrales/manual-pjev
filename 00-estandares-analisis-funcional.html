Estándar de Análisis Funcional

Estructura obligatoria para documentar el levantamiento de requerimientos y diseño de nuevos módulos.

1 Objetivo

El Análisis Funcional es la piedra angular del desarrollo. Define el "Qué" y el "Por qué" de un módulo antes de escribir una sola línea de código. Ningún módulo debe desarrollarse sin que su Análisis Funcional correspondiente haya sido redactado, revisado y autorizado.

Todos los archivos de Análisis Funcional deben nombrarse con la convención 0X.Y.Z-nombre-modulo.html (por ejemplo, 01.1.1-entidad.html) y deben seguir estrictamente la estructura detallada a continuación.

2 Estructura Obligatoria de 9 Puntos

1. Descripción General

Un resumen de 1-2 párrafos que explique el propósito del módulo en un lenguaje no técnico y orientado al negocio.

2. Marco Normativo

Tabla obligatoria que vincula el módulo con la ley (LGCG, CONAC, etc.). Columnas: Norma, Artículo / Punto, Obligación que genera.

3. Roles que intervienen

Tabla de accesos y permisos. Columnas: Rol, Acción permitida, Restricción.

4. Modelo de Datos

Tabla con los campos físicos en la base de datos, marcando llaves (PK, FK) y requeridos. Columnas: Campo, Tipo, Descripción.

5. Stored Procedures (SPs)

Se deben definir obligatoriamente las funciones fn_leer_[modulo] y fn_crud_[modulo].

  • Los parámetros de entrada (IN) deben denotarse con el prefijo @ (ej. @usuario, @accion, @parametro).
  • Los valores devueltos (OUT) o atributos JSON deben denotarse entre llaves {} (ej. {dato}, {error}).
  • El orden de los parámetros base siempre es: @usuario, @ejercicio, @accion.

6. Servicios REST

Endpoints que el Frontend consumirá. Columnas: Método (GET, POST, PUT, DELETE), Endpoint, Body / Params, Descripción.

7. Diseño de Interfaces (UI / Wireframes)

Todo análisis debe incluir wireframes (dibujados en SVG embebido) mostrando las dos fases del estándar Frontend en 4 pestañas interactivas:

  • pantalla_leer con @parametros: Pantalla base o grid mapeando los bindings de variables (ej. mostrando {campo} en la tabla).
  • pantalla_leer con valores: La misma pantalla pero ilustrada con datos de ejemplo reales o mockups para el usuario final.
  • pantalla_abc con @parametros: Formulario de captura/edición (modal/ventana) indicando dónde se captura cada @parametro del SP.
  • pantalla_abc con valores: Formulario de captura ilustrado con datos de ejemplo.

8. Reglas de Negocio Críticas

El corazón de la lógica funcional. Debe presentarse en una tabla con las siguientes 5 columnas estrictas:

1. Código: (ej. RN-01) 2. Regla: (Descripción clara de la regla) 3. Consecuencia: (Qué sucede si se rompe, ej. Error en pantalla) 4. Dónde se valida: (SP, Frontend Zod, Trigger, Procesos Core) 5. Fuente (Normativa / Técnica): (Fundamento explícito, ej. "Ley General de Cont. Gubernamental Art. 18" o "Restricción técnica del modelo de datos")

9. Flujos de Integración

Diagrama de texto que explica cómo este módulo interactúa con otros (ej. "Entidad afecta a -> Pólizas Automáticas").
