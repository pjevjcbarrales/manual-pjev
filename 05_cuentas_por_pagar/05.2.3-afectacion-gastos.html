05.2.3

Afectación de Gastos (Órdenes de Pago)

Módulo transaccional de Cuentas por Pagar encargado de registrar las facturas (CFDI), clasificar el gasto presupuestalmente y gestionar el flujo de autorizaciones para su pago.

Fase 2: Ejecución del Gasto Tipo: Formulario Compuesto (1:N) Ruta: cxp / registros / afectacion_gasto
1 Descripción General

La opción de Afectación de Gastos es el núcleo operativo del Departamento de Tesorería. Sustituye al antiguo esquema basado exclusivamente en el llenado manual del "Anexo 9". Este módulo permite registrar un trámite de pago capturando uno o múltiples archivos XML (CFDI) asociados a un único beneficiario (Proveedor, Empleado, Unidad Administrativa o Tercero).

Una vez capturada la cabecera y vinculadas las facturas, el analista asigna el gasto a la estructura programática correspondiente (Organización, Función, Programa, Proyecto y Partida - OGTO) en el grid de detalles, validando en tiempo real que exista disponibilidad presupuestal suficiente antes de enviar el trámite a la cadena de firmas y autorización.

2 Marco Normativo
NormaArtículo / PuntoObligación que genera
Ley General de Contabilidad Gubernamental (LGCG) Art. 4 Fracc. XVI y Art. 9 Asegurar el registro armónico de las etapas del gasto. Una afectación confirmada debe detonar automáticamente los momentos contables de Comprometido y Devengado.
Reglamento Interior del PJEV Manual de Procedimientos (Recursos Financieros) Obliga a la generación de la "Solicitud-comprobación de recurso" (Orden de Pago) mediante el sistema institucional para verificar el techo financiero.
SAT - Anexo 20 Reglas de validación CFDI Extracción y validación fidedigna del UUID, RFC del receptor/emisor y desglose de impuestos retenidos para el entero de obligaciones fiscales.
3 Roles que intervienen
RolAcción permitidaRestricción
Analista de Tesorería / Capturista Crear la afectación, cargar CFDIs, clasificar las partidas (OGTO) y enviar a validación. Solo puede editar afectaciones en estatus CAPTURA o RECHAZADO. No autoriza pagos.
Jefe de Control Presupuestal Valida que la partida, proyecto y suficiencia presupuestal sean correctas. Pasa el estatus a VALIDADO. No puede alterar los XML cargados, solo rechazar o aprobar la clasificación.
Titular / Director Administrativo Autoriza la orden de pago (Estatus AUTORIZADO) firmando digitalmente la afectación. Requiere que el trámite haya pasado por Control Presupuestal.
Analista de Cuentas por Pagar Programa el pago y marca el trámite como PAGADO una vez transferido el recurso en bancos. Solo puede actuar sobre registros AUTORIZADOS.
4 Modelo de Datos

La persistencia del módulo se diseñó para operar grandes volúmenes de comprobaciones sin saturar la tabla principal. La cabecera se aloja en cxp.afectaciones_gasto particionada por ejercicio fiscal y garantiza la unicidad de beneficiarios, delegando las N facturas (CFDI) a la tabla hija _cfdi y la validación presupuestal a la tabla _detalles.

CREATE OR REPLACE FUNCTION cxp.fn_leer_afectacion_encabezado(
    p_ejercicio_fiscal INT,
    p_ag_id UUID DEFAULT NULL,
    p_folio VARCHAR DEFAULT NULL,
    p_concepto VARCHAR DEFAULT NULL,
    p_f_proveedor_id INT DEFAULT NULL,
    p_f_centro_costo_id INT DEFAULT NULL,
    p_uuid_cfdi VARCHAR DEFAULT NULL,
    p_estatus_workflow VARCHAR DEFAULT NULL,
    p_es_rh BOOLEAN DEFAULT FALSE
) RETURNS jsonb LANGUAGE plpgsql AS $$
DECLARE v_resultado jsonb;
BEGIN
    SELECT COALESCE(jsonb_agg(row_to_json(t)), '[]'::jsonb) INTO v_resultado
    FROM (
        SELECT 
            a.ag_id, a.ejercicio_fiscal, a.folio, a.tipo_pago, a.afecta_presupuesto, a.cuenta_contable_abono,
            a.f_centro_costo_id, a.concepto, a.fecha_elaboracion, a.estatus_workflow,
            a.f_proveedor_id, a.f_empleado_id, a.f_ua_id, a.f_tercero_id, a.f_tercero_inst_id
        FROM cxp.afectaciones_gasto a
        WHERE a.ejercicio_fiscal = p_ejercicio_fiscal
          AND a.estatus_registro = 1
          AND (p_ag_id IS NULL OR a.ag_id = p_ag_id)
          AND (p_folio IS NULL OR a.folio ILIKE '%' || p_folio || '%')
          AND (p_concepto IS NULL OR a.concepto ILIKE '%' || p_concepto || '%')
          AND (p_f_proveedor_id IS NULL OR a.f_proveedor_id = p_f_proveedor_id)
          AND (p_f_centro_costo_id IS NULL OR a.f_centro_costo_id = p_f_centro_costo_id)
          AND (p_estatus_workflow IS NULL OR a.estatus_workflow = p_estatus_workflow)
          -- Segregación Capítulo 1000 (Nómina)
          AND (
              (p_es_rh = TRUE AND a.tipo_pago = 4) OR
              (p_es_rh = FALSE AND a.tipo_pago <> 4)
          )
          AND (p_uuid_cfdi IS NULL OR EXISTS (
              SELECT 1 FROM cxp.afectaciones_gasto_cfdi c WHERE c.f_ag_id = a.ag_id AND c.uuid_cfdi ILIKE '%' || p_uuid_cfdi || '%'
          ))
    ) t;
    RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'OK', 'data', v_resultado);
END;
$$;

        INSERT INTO cxp.afectaciones_gasto(

        ) RETURNING ag_id INTO v_id_gen;
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Alta exitosa', 'id_generado', v_id_gen);

    ELSIF p_accion = 2 THEN
        -- Re-calcular cuenta en actualización

        UPDATE cxp.afectaciones_gasto
        SET tipo_pago = p_tipo_pago, cuenta_contable_abono = v_cuenta_abono, afecta_presupuesto = p_afecta_presupuesto,

CREATE OR REPLACE FUNCTION cxp.fn_abc_afectacion_detalles(
    p_accion INT,
    p_agd_id UUID DEFAULT NULL,
    p_f_ag_id UUID DEFAULT NULL,
    p_f_centro_costo_id INT DEFAULT NULL,
    p_f_funcion_id INT DEFAULT NULL,
    p_f_programa_id INT DEFAULT NULL,
    p_f_subprograma_id INT DEFAULT NULL,
    p_f_proyecto_id INT DEFAULT NULL,
    p_f_cog_id INT DEFAULT NULL,
    p_f_fuente_fin_id INT DEFAULT NULL,
    p_importe_solicitado NUMERIC DEFAULT NULL,
    p_usuario VARCHAR DEFAULT 'SISTEMA'
) RETURNS jsonb LANGUAGE plpgsql AS $$
BEGIN
    IF p_accion = 1 THEN
        -- Validación Presupuestal Simulada
        INSERT INTO cxp.afectaciones_gasto_detalles(
            f_ag_id, f_centro_costo_id, f_funcion_id, f_programa_id, f_subprograma_id,
            f_proyecto_id, f_cog_id, f_fuente_fin_id, importe_solicitado, disponibilidad_presupuestal, usuario_c
        ) VALUES (
            p_f_ag_id, p_f_centro_costo_id, p_f_funcion_id, p_f_programa_id, p_f_subprograma_id,
            p_f_proyecto_id, p_f_cog_id, p_f_fuente_fin_id, p_importe_solicitado, 0, p_usuario
        );
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Partida asignada con éxito');
    
    ELSIF p_accion = 2 THEN
        UPDATE cxp.afectaciones_gasto_detalles
        SET f_centro_costo_id = p_f_centro_costo_id, f_funcion_id = p_f_funcion_id,

        WHERE agd_id = p_agd_id;
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Partida actualizada');
    
    ELSIF p_accion = 3 THEN
        DELETE FROM cxp.afectaciones_gasto_detalles WHERE agd_id = p_agd_id;
        RETURN jsonb_build_object('cod_resultado', 1, 'msg_resultado', 'Partida eliminada');
    END IF;
END;
$$;
6 Servicios REST (Endpoints NestJS)

El backend (NestJS) expone los siguientes endpoints bajo el controlador AfectacionesGastoController. Cada servicio valida el payload mediante DTOs (Data Transfer Objects) y consume los Stored Procedures definidos en la sección anterior utilizando Prisma $queryRaw.

9 Reglas Críticas de Negocio

Las siguientes reglas rigen la integridad de la información y la lógica de validación tanto en el Backend (Stored Procedures) como en el Frontend (Formulario de Captura). No deben ser omitidas por ningún desarrollador.

Regla / Restricción Nivel Comportamiento esperado (Validación)
RN-01: Bloqueo por Estatus Workflow Backend / UI La edición y eliminación (física o lógica) de una cabecera y sus partidas solo está permitida si estatus_workflow es 'CAPTURA' o 'RECHAZADO'. Si está en 'VALIDADO', 'AUTORIZADO' o 'PAGADO', la base de datos abortará la transacción.
RN-02: Determinación del Beneficiario Único Backend / UI Dependiendo del tipo_pago seleccionado, solo se debe guardar el ID en un único campo de beneficiario, estableciendo los demás en nulo. Ejemplo: Si tipo_pago = 1 (Proveedores), f_proveedor_id es obligatorio y f_empleado_id / f_ua_id deben ser NULOS obligatoriamente.
RN-03: Derivación de la Estructura Programática Frontend El usuario analista no debe capturar manualmente las claves Programáticas (Programa, Subprograma, Proyecto, Fuente y Tipo de Gasto). El frontend las debe recuperar del cruce paramétrico entre el Centro de Costo (Organización) seleccionado en la cabecera, y la Partida Específica seleccionada en el grid.
RN-04: Tope de Disponibilidad Presupuestal Backend (SP) Al invocar fn_abc_afectacion_detalles (Inserción de partida), el SP debe consultar la vista o servicio de suficiencia SAFPOJ. Si el @p_importe_solicitado excede el saldo de esa estructura programática, se devuelve cod_resultado = -1 ("Saldo insuficiente") y se hace ROLLBACK de esa partida.
RN-05: Modificación de Partidas Restrictiva Backend (SP) Para modificar un monto en el grid de partidas (Update), se recomienda eliminar la partida y volverla a insertar para disparar limpiamente la re-validación del saldo presupuestal, evitando desfases aritméticos en la concurrencia de presupuestos.
RN-06: Aislamiento del Ejercicio Fiscal Frontend / BD El ejercicio_fiscal jamás se solicita al capturista. El backend lo inyecta directamente desde la variable de entorno o sesión activa. La base de datos rechaza cualquier manipulación ajena gracias a su particionamiento nativo.
RN-07: Relación con el Módulo CFDI (Fase 2) Arquitectura Aunque la selección de XMLs / UUID se pospone en la UI, la sumatoria del monto de las partidas capturadas deberá coincidir en el futuro de manera exacta con el importe_total neto de los comprobantes asociados a la afectación.
RN-08: Segregación del Capítulo 1000 (Nómina / RH) BD / Frontend Los usuarios pertenecientes al área de Recursos Humanos (RH) tendrán acceso exclusivo para visualizar, capturar y gestionar las afectaciones correspondientes a Nómina (tipo_pago = 4 / Capítulo 1000). El resto de los usuarios (Tesorería, etc.) solo podrán gestionar los tipos de pago ajenos a RH (Proveedores, Viáticos, etc.), ocultándose mutuamente la información para garantizar la confidencialidad de la nómina.
10 Puntos de Integración (Interoperabilidad)

El Módulo de Afectación de Gastos no opera de manera aislada. Funciona como el engrane principal que conecta el presupuesto autorizado con la ejecución del pago en tesorería, interactuando con los siguientes ecosistemas del SIAF-PJEV:

Módulo Origen / Destino Tipo de Interacción Descripción del flujo de datos
Sistema de Presupuesto (SAFPOJ) Lectura en Tiempo Real Durante la captura de detalles, se consume el techo financiero (Disponibilidad Presupuestal) del cruce Programático-Estructural. Sin esta validación, la base de datos no permite afectar la partida.
Módulo de Facturación (CFDI) Lectura y Vinculación (Fase 2) El módulo extrae los datos duros de los XML timbrados (UUID, RFC, Subtotal, IVA, Retenciones de ISR/IVA) y los vincula mediante la tabla _cfdi para evitar la recaptura de importes.
Tesorería y Bancos (Dispersión) Salida de Datos (Trigger) Una vez que la Afectación de Gasto llega al estatus 'AUTORIZADO', se convierte en el insumo directo (Orden de Pago) para que Bancos genere los layouts bancarios (txt). Al confirmarse la transferencia, Bancos actualiza la afectación a 'PAGADO'.
Gestión de Proveedores / RRHH Lectura de Catálogos El módulo consume el Padrón de Proveedores (para trámites de compras) y la plantilla de RRHH (para viáticos o comprobaciones de nómina) garantizando que los pagos se emitan a entidades legalmente registradas.
11 Impacto Contable (Armonización CONAC)

Conforme a la Ley General de Contabilidad Gubernamental, este módulo es el principal detonador de las pólizas automatizadas de egresos. La aprobación de una afectación de gasto transita por los siguientes momentos contables y registros de libro diario:

Evento en el Sistema Impacto Presupuestal Generación de Póliza Contable (Automática)
El Jefe de Control Presupuestal cambia el estatus a VALIDADO Se registra el 3er Tiempo Contable (Gasto Comprometido). El sistema aparta el presupuesto para cada cuenta de gasto (Partida), disminuyendo el disponible y aumentando el comprometido. No genera póliza contable, únicamente afecta los saldos de los clasificadores presupuestales.
El Director Administrativo firma y cambia el estatus a AUTORIZADO (Orden de Pago) El sistema mueve automáticamente el presupuesto del tercer tiempo contable hasta el Gasto Ejercido por cada cuenta de gasto (transitándolo por el Devengado y Ejercido de manera secuencial). Genera Póliza de Diario (Devengado):
[+] Cargo: Cuentas de Gastos (Ej. 5000 - Gastos de Funcionamiento) por el Subtotal.
[-] Abono: Pasivo Circulante (Utiliza la `cuenta_contable_abono` generada en la cabecera) por el Importe Neto.
[-] Abono: Pasivo por Retenciones (Ej. 2117 - ISR / IVA Retenido).
El área de Tesorería transfiere el recurso y cambia el estatus a PAGADO Se registra el momento del Gasto Pagado. Concluye el ciclo presupuestal del egreso. Genera Póliza de Egreso (Pago):
[+] Cargo: Cancelación del Pasivo por el Importe Neto.
[-] Abono: Bancos / Efectivo y Equivalentes (Ej. 1112 - Bancos Tesorería).
